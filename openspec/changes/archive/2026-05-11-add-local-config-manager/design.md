## Context

`F:/Homework/memoflow_config` 是 MemoFlow 远端配置的独立仓库。当前源文件被拆分为：

- `update/manifest.json`: 全局版本配置、legacy notice 配置、announcement 索引。
- `update/announcements/*.json`: 每个版本更新公告一个文件。
- `update/donors.json`: 捐赠者列表。
- `update/assets/*`: 捐赠者头像等静态资源。

`.github/scripts/build_update_config.py` 负责把这些源文件合并为 `dist/update/latest.json`。App 侧已引入 `schema_version: 3` announcement delivery contract，并保持 legacy config 兼容。这个本地工具应服务于配置仓库，不应成为 Flutter runtime 的一部分。

当前架构阶段是 `evolve_modularity`。本变更不触碰 `memos_flutter_app/lib` 的 `state -> features`, `application -> features`, or `core -> higher-layer` coupling hotspots；依赖方向只发生在 `memoflow_config/config` 工具内部：

```text
Browser UI
    |
    v
localhost API service
    |
    v
update/*.json source files
    |
    v
build_update_config.py
    |
    v
dist/update/latest.json
```

The local service is the write owner. The browser UI never writes files directly once the localhost tool is used.

## Goals / Non-Goals

**Goals:**

- Provide a local browser UI for editing update announcements, historical release notes, v3 notices, v3 updates, and donors.
- Preserve split source files as the only editable source of truth.
- Keep `dist/update/latest.json` generated by the existing build script.
- Support v3 authoring while keeping old app versions compatible through explicit legacy sync paths.
- Run validation and build commands from the local service.
- Keep all new code in `F:/Homework/memoflow_config/config/` unless a narrowly-scoped build-script change is required.
- Use a local service boundary so file writes, path validation, and command execution have one clear owner.

**Non-Goals:**

- No hosted admin console.
- No authentication, multi-user collaboration, or cloud database.
- No changes to app runtime announcement policy.
- No changes to commercial/private extension behavior.
- No direct editing of `dist/update/latest.json`.
- No automatic Git commit, push, or publish in the first implementation. The tool may show status and leave Git sync as an explicit user step.

## Decisions

### Decision 1: Build a localhost Python service plus static browser UI

The tool will live under `F:/Homework/memoflow_config/config/` with:

- `server.py`
- `manager.html`
- `manager.js`
- `manager.css`

`server.py` will bind to `127.0.0.1` and serve the UI plus JSON API endpoints. Python is already required by the existing config build workflow, and a stdlib-based server avoids adding Node/npm or frontend build dependencies.

Alternative considered: pure `file://` HTML using browser file-system APIs. Rejected because command execution, validation, build generation, and asset upload are awkward and browser-dependent. A local service provides a stable write and command boundary.

### Decision 2: Treat `update/` as source and `dist/` as output

All edits target source files:

- `update/manifest.json`
- `update/announcements/*.json`
- `update/donors.json`
- `update/assets/*`

The tool may call:

```bash
python .github/scripts/build_update_config.py --root update --validate-only
python .github/scripts/build_update_config.py --root update --output dist/update/latest.json
```

Alternative considered: edit `dist/update/latest.json` directly. Rejected because CI and publish workflows regenerate it, and direct edits would drift from source files.

### Decision 3: Store v3 `notices[]` and `updates[]` in `manifest.json`

The current build script deep-copies `manifest.json` and removes only split-source bookkeeping fields such as `announcement_ids`, `latest_announcement_id`, `donors_file`, and `announcement_tag_index`. Therefore `notices[]` and `updates[]` can be authored in `manifest.json` and carried into `latest.json` without changing the split file layout.

Alternative considered: add `update/notices.json` and `update/updates.json`. Rejected for the first version because it requires broader build-script, CI, and documentation changes. It remains a possible future cleanup if the arrays become large.

### Decision 4: Use explicit legacy compatibility sync

The UI will distinguish v3 authoring from legacy compatibility:

- v3 `notices[]` are the canonical new notification model.
- One v3 notice can be marked as "sync to legacy notice"; saving writes `manifest.notice_enabled = true` and writes a compatible `manifest.notice` block.
- If no notice is selected for legacy sync, the tool leaves `notice_enabled` and `notice` under user-visible control, with a safe action to disable legacy notice.
- v3 `updates[]` are the canonical new update candidates.
- A selected primary update candidate can sync to `manifest.version_info.<platform>` so older app versions continue seeing update prompts.

Alternative considered: automatically mirror every public v3 item into legacy fields. Rejected because legacy fields can represent only one notice and platform-scoped version blocks, so automatic mirroring could publish unintended content.

### Decision 5: Manage update announcements through existing announcement files

The update-announcement editor will load `manifest.latest_announcement_id`, then edit the matching `update/announcements/<id>.json`. Creating a new update announcement will:

- Generate a numeric string id compatible with the existing worker and build script, using UTC timestamp plus a stable suffix.
- Create `update/announcements/<id>.json`.
- Add the id to `manifest.announcement_ids`.
- Set `manifest.latest_announcement_id`.
- Update `manifest.announcement_tag_index["v<version>"]` when a release tag is provided.

Historical announcements are displayed from `manifest.announcement_ids` and sorted using the build script's numeric-id behavior. Editing historical records should be possible only through an explicit edit action so accidental release-note drift is less likely.

### Decision 6: Validate before writing risky changes and after saving

The service will parse JSON through structured JSON APIs and write UTF-8 JSON with stable formatting. It will reject path traversal and only allow writes under the configured repository root.

Validation layers:

- Schema-shape validation inside the local service before saving.
- Existing split-config validation through `build_update_config.py --validate-only`.
- Optional production-safety diagnostics mirroring app-side `validate_announcement_config.dart` rules for v3 `notices[]` and `updates[]`.

The first implementation can surface diagnostics from Python validation and build script output in the UI.

### Decision 7: Keep UI previews local and deterministic

The browser UI will render local previews for:

- update announcement dialog content
- notification notice content
- release note groups
- donor cards
- update candidate summaries

Preview is not a substitute for app Debug preview. It is an authoring aid. The generated JSON remains the contract that app-side preview and runtime consume.

## Risks / Trade-offs

- [Risk] Local service writes the wrong path or corrupts source JSON. -> Mitigation: resolve paths against repository root, reject traversal, use atomic write through a temporary file and replacement, and parse after write.
- [Risk] v3 fields in `manifest.json` make the file too large. -> Mitigation: first version favors minimal build-script changes; if scale becomes painful, split `notices` / `updates` into dedicated files later.
- [Risk] Legacy sync accidentally publishes one notice to old clients. -> Mitigation: legacy sync is opt-in and visible; disabling legacy notice is a separate explicit action.
- [Risk] Build script currently validates split announcements but not full v3 production safety. -> Mitigation: add local service validation for v3 fields and consider porting rules into the config repo if needed.
- [Risk] Browser UI and app dialog preview diverge visually. -> Mitigation: preview focuses on content, targeting, and compatibility shape rather than exact Flutter rendering.
- [Risk] Running a local service increases local attack surface. -> Mitigation: bind only to `127.0.0.1`, avoid arbitrary shell command endpoints, and expose only fixed validation/build actions.

## Migration Plan

1. Add the `config/` local tool files without changing existing config source files.
2. Implement read-only loading first: manifest, announcements, donors, and generated preview summary.
3. Add write actions for donors and announcement files with validation.
4. Add v3 `notices[]` and `updates[]` editors in `manifest.json`.
5. Add explicit legacy sync controls for selected notice and primary update candidate.
6. Add build/validate actions that call the existing Python script.
7. Verify generated `dist/update/latest.json` preserves legacy fields and includes v3 arrays.

Rollback:

- Remove or ignore `config/` tool files; existing manual JSON workflow remains valid.
- If a saved config is wrong, restore prior Git state for `update/manifest.json`, `update/announcements/*.json`, and `update/donors.json`.
- For published notice rollback, set affected v3 items to `archived` and disable legacy `notice_enabled` if needed, then rebuild.

## Open Questions

- Should production-safety validation rules be duplicated in `config/server.py` immediately, or should the first version rely on build validation plus visible warnings?
- Should the first version upload/copy avatar files into `update/assets/`, or only manage donor records with existing/public avatar URLs?
- Should the tool show Git status only, or include fixed local commands such as `git diff -- update` without commit/push support?
