## Context

The app uses Slang YAML resources under `memos_flutter_app/lib/i18n/` and generated accessors in `strings.g.dart`. Runtime language choice flows through `AppLanguage`, `DevicePreferences.language`, `appLocaleForLanguage`, and `LocaleSettings.setLocale`. Settings mostly derives language options from `AppLanguage.values`, while onboarding has an explicit language list and switch-based labels. The image editor plugin has a separate in-memory i18n map.

The active architecture phase is `evolve_modularity` with a recorded modularity score of `4/10`. This change touches localization seams but should not touch the known reverse-dependency hotspots. Dependency direction before and after remains UI/features -> state/data/core/i18n and core -> data/i18n only through the existing `app_localization.dart` seam; no new `state -> features`, `application -> features`, or `core -> higher layer` dependency should be introduced.

## Goals / Non-Goals

**Goals:**

- Add Brazilian Portuguese (`pt-BR`) as a first-class selectable app language.
- Generate a complete `strings_pt-BR.i18n.yaml` from the English source using AI-assisted translation.
- Preserve all Slang interpolation placeholders, variables, key names, namespace structure, and technical identifiers.
- Ensure system/device Portuguese locales resolve to the Portuguese (Brazil) app locale.
- Cover Slang resources and the image editor plugin i18n surface.
- Add focused tests or guardrails that verify locale mapping, generated locale support, and key Portuguese labels.

**Non-Goals:**

- Do not add Portuguese (Portugal) as a separate locale in this change.
- Do not change app storage schemas, sync behavior, server API compatibility, private extension seams, or commercial behavior.
- Do not restructure the whole localization architecture beyond small guardrails needed for this locale.
- Do not manually edit `strings.g.dart`; it must be regenerated by the existing Slang workflow.

## Decisions

### Decision 1: Use `pt-BR` as the locale identity

Use `strings_pt-BR.i18n.yaml`, `AppLanguage.ptBr`, and the generated `AppLocale.ptBr` value. Language labels should use `Portuguese (Brazil)` in English and `Português (Brasil)` natively.

Alternatives considered:

- Generic `pt`: simpler but loses the user's explicit Brazilian Portuguese intent.
- `pt-PT`: incorrect target audience for this change.
- Both `pt-BR` and `pt-PT`: higher translation/review scope than requested.

### Decision 2: Translate from English source with placeholder preservation

Generate the new Portuguese resource from `strings.i18n.yaml` because it is the Slang base locale and avoids translating from already localized strings. The translation process must preserve YAML structure and placeholders such as `$version`, `$e`, and function parameters exactly.

Alternatives considered:

- Copy English as a placeholder: lower risk but does not satisfy the requested translation.
- Translate from Chinese resources: risks second-order translation drift and existing encoding display issues.
- Manually translate every string without AI assistance: better quality ceiling but too slow for the requested path.

### Decision 3: Map any Portuguese system language to `pt-BR`

`appLanguageFromLocale` and `appLocaleForLanguage` should treat `pt` and `pt-BR` as Portuguese (Brazil). Since the app will only ship one Portuguese variant, other Portuguese regional device locales such as `pt-PT` should prefer `pt-BR` over falling back to English.

Alternatives considered:

- Only map exact `pt-BR`: more regionally precise but surprising for Portuguese-language users on other regions.
- Keep Portuguese system users on English unless exact match: preserves regional purity but weakens localization discovery.

### Decision 4: Integrate through existing language seams

Add `AppLanguage.ptBr` and extend existing switches/lists rather than introducing a new language registry in this change. Settings can continue to use `AppLanguage.values`; onboarding must add the new enum value to its explicit list and title/subtitle switches.

Alternatives considered:

- Introduce a full language registry to remove onboarding switches: architecturally attractive, but larger than this localization change and risks touching more feature code during `evolve_modularity`.
- Hard-code Portuguese only in settings: incomplete because onboarding is the first language selection surface.

### Decision 5: Add a Portuguese image editor i18n map

The image editor plugin does not use Slang resources, so it should receive a small `pt-BR` map for the existing editor labels. If a plugin label is not covered, existing plugin fallback behavior may still show English.

Alternatives considered:

- Leave image editor on English: inconsistent with a first-class app locale.
- Move plugin strings into Slang now: cleaner long-term, but broader and unnecessary for adding this locale.

### Decision 6: Add focused localization guardrails

Add tests for key `pt-BR` generated labels and locale mapping. This improves modularity checklist item `8` without changing dependency direction. The tests should avoid large snapshot assertions and focus on stable labels, supported locale membership, and system mapping behavior.

Alternatives considered:

- Rely only on `dart run slang`: catches structural issues but not app-level mapping.
- Add broad UI golden tests: expensive and brittle for a language-resource change.

## Risks / Trade-offs

- AI translation quality may be awkward or inconsistent -> Mitigate with a small terminology list, key label tests, and later human review.
- Placeholder or YAML structure corruption could break generation -> Mitigate by translating only values, preserving keys/variables exactly, then running `dart run slang`.
- Generated `strings.g.dart` will be large -> Mitigate by keeping manual edits in YAML only and committing generated output from the standard tool.
- `pt-PT` users will see Brazilian Portuguese -> Mitigate by documenting this fallback as intentional until a separate `pt-PT` locale exists.
- Existing language enum persistence stores enum names -> Mitigate by adding only a new enum value; rollback to a build without `ptBr` would fall back via existing unknown-value parsing.

## Migration Plan

1. Add Portuguese labels to existing locale YAML files in both `languages` and `legacy.app_language`.
2. Add `strings_pt-BR.i18n.yaml` translated from the English base resource.
3. Extend `AppLanguage`, locale mapping, onboarding language selection, and image editor i18n handling.
4. Regenerate `strings.g.dart` from `memos_flutter_app` with `dart run slang`.
5. Run focused i18n tests, then broader `flutter analyze` and `flutter test` before PR.

Rollback is a normal code revert. No user data migration is required; if a user preference stores `ptBr` and an older build reads it, existing enum parsing should fall back to the default language behavior.

## Open Questions

- None blocking. A later change may add separate `pt-PT` support if users request European Portuguese.
